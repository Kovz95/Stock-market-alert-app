# Questions: Custom Discord Condition Channels (03)

## Resolved

| # | Question | Resolution |
|---|----------|------------|
| R1 | What feature is being restored? | The "Custom Discord Channels" tab that exists in `pages/Discord_Management.py` (Streamlit) but is absent from the new React UI at `apps/ui/web/app/discord/`. It lets a user create a Discord channel that receives alerts matching a specific condition string (e.g. `RSI(14)[-1] < 30`, `Close[-1] > 100`, or the special keyword `price_level`). |
| R2 | Does backend routing still support custom channels? | Yes. `src/services/discord_routing.py` reads `custom_discord_channels.json` and, in `get_custom_channels_for_alert()`, sends a copy of every alert whose condition matches. Nothing changed in the alert pipeline — only the management UI was lost. |
| R3 | What shape does a custom channel entry have today? | `{ webhook_url, description, condition, enabled, created (ISO), channel_name }` stored under a unique display name as the map key. (See `pages/Discord_Management.py:617–624`.) |
| R4 | Where in the new UI does this belong? | A new page at `/discord/custom` (sidebar group "Discord"), reusing `DiscordChannelForm` / shadcn primitives already in `apps/ui/web/app/discord/_components/`. |
| R5 | What condition presets did the old UI expose? | "Custom (enter manually)", "Price Level (any price comparison)", "RSI - Oversold", "RSI - Overbought", "MACD Crossover", "Moving Average Cross", "Bollinger Bands", "Volume Breakout", "HARSI Flip" — see `pages/Discord_Management.py:520–596`. |
| R6 | What does the `price_level` special keyword do? | It matches **any** alert condition that compares `Close`/`Open`/`High`/`Low` to a numeric literal (regex in `is_price_level_condition()` in `src/services/discord_routing.py:254`). Unlike other conditions (exact-match after normalization), `price_level` is a category match. The new UI must surface this distinction. |
| R7 | Is condition matching case/whitespace sensitive? | No. `normalize_condition()` in `discord_routing.py` lowercases and strips whitespace around operators before comparing. Exact spelling after normalization is what matters — the UI should not silently transform expressions. |
| R8 | Which proto service owns the new RPCs? | `DiscordConfigService` in `proto/discord/v1/discord.proto`. Add new message types and RPCs there; do not create a new service. |
| R9 | Do custom channels interact with timeframe splits (hourly/daily/weekly)? | No — in the current Python router, custom channels match regardless of alert timeframe. This spec preserves that behavior. Per-timeframe custom channels can be a follow-up. |
| R10 | Where is `custom_discord_channels.json` stored? | It is loaded via `load_document("custom_discord_channels", …)` in `discord_routing.py:210` with a `fallback_path` of `custom_discord_channels.json` at the repo root. The Go service must read/write through the same logical document so the Python scheduler and Go scheduler port both see the same config. |

## Open

| # | Question | Impact |
|---|----------|--------|
| O1 | **Storage backend for the Go `DiscordConfigService`.** The existing Python router reads `custom_discord_channels.json` via the `document_store` abstraction backed by `app_documents` in Postgres (with JSON file fallback). Should the Go service read/write the same `app_documents.custom_discord_channels` row, or should we introduce a dedicated `custom_discord_channels` Postgres table? Default in this spec: **reuse the existing `app_documents` document** so the Python and Go schedulers remain in sync without migration. If a dedicated table is preferred, revise before Task 02. |
| O2 | **Condition input UX — reuse `ConditionBuilder` or keep the old preset dropdown?** The new React `/alerts/add` page has a full-featured `ConditionBuilder` that emits canonical expression strings. Options: (a) reuse `ConditionBuilder` + add a "Special: Price Level (any)" toggle for the `price_level` keyword, (b) ship a simpler preset dropdown matching the Streamlit page (RSI / MACD / MA Cross / BB / Volume / HARSI / Custom / Price Level). Default in this spec: **(a) reuse `ConditionBuilder`** with a top-level radio toggle for `price_level`-vs-explicit-condition, because exact-string matching makes consistency with the alert builder critical (the same canonical expression must appear on both sides). If parity with the old preset list is the priority, switch to (b) before Task 05. |
| O3 | **Should saved condition strings be validated server-side before persisting?** The Python side just stores whatever string the user submitted. We could run the Go `expr` parser on the string when creating/updating to catch typos. Default in this spec: **validate via the `expr` package** in the Go service — reject on parse failure with a clear error. Cost: one import; benefit: no silent "channel saves but never matches" bugs. |
| O4 | **Deletion — soft-delete or hard-delete?** Old UI hard-deletes. Default: **hard-delete** to match. Enabled/disabled toggle already covers the "temporarily off" case. |
| O5 | **Test button parity.** Streamlit sends an embedded test message showing the channel name + stored condition. Default: mirror this behavior via a new `SendCustomChannelTestMessage` RPC. Alternative: reuse the existing `SendHourlyTestMessage` shape — but that resolves by channel key (economy), not by custom-channel id, so a new RPC is cleaner. |
